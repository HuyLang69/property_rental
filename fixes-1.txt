2. LOGIC ERROR: ListingController.php (Lines 42-48)
   ─────────────────────────────────────────────────────────────────────────
   File: app/Http/Controllers/Listingcontroller.php
   Location: index() method - sorting logic
   
   Error: Match statement return values were discarded. The match block did not
   properly apply sorting to the query builder.
   
   Before:
   $sort = $request->get('sort', 'newest');
   match ($sort) {
       'price_asc'  => $query->orderBy('price_per_night', 'asc'),
       'price_desc' => $query->orderBy('price_per_night', 'desc'),
       'rating'     => $query->withAvg('reviews', 'rating')->orderByDesc('reviews_avg_rating'),
       default      => $query->orderBy('created_at', 'desc'),
   };
   // ↑ Return values were discarded; sorting didn't apply reliably
   
   After:
   $sort = $request->get('sort', 'newest');
   if ($sort === 'price_asc') {
       $query->orderBy('price_per_night', 'asc');
   } elseif ($sort === 'price_desc') {
       $query->orderBy('price_per_night', 'desc');
   } elseif ($sort === 'rating') {
       $query->withAvg('reviews', 'rating')->orderByDesc('reviews_avg_rating');
   } else {
       $query->orderBy('created_at', 'desc');
   }
   
   Impact: Listings sorting feature (price ascending, price descending, rating)
   would not work properly. Users would not see listings sorted as selected.


3. CRITICAL LOGIC ERROR: BookingController.php (Lines 35-44 and Lines 100-109)
   ─────────────────────────────────────────────────────────────────────────
   File: app/Http/Controllers/BookingController.php
   Location: create() method AND store() method - date conflict detection
   
   Error: Date overlap detection used inclusive whereBetween() which caused
   false positive conflicts when bookings were adjacent (e.g., existing checkout
   on Feb 10 conflicting with new checkin on Feb 10).
   
   Before (in both create() and store() methods):
   $conflict = Booking::where('listing_id', $listing->id)
       ->where('status', 'confirmed')
       ->where(function ($q) use ($request) {
           $q->whereBetween('check_in',  [$request->check_in, $request->check_out])
             ->orWhereBetween('check_out', [$request->check_in, $request->check_out])
             ->orWhere(function ($q) use ($request) {
                 $q->where('check_in',  '<=', $request->check_in)
                   ->where('check_out', '>=', $request->check_out);
             });
       })->exists();
   // ↑ whereBetween() is inclusive on both ends, causing false positives
   
   After:
   $conflict = Booking::where('listing_id', $listing->id)
       ->where('status', 'confirmed')
       ->where(function ($q) use ($request) {
           $q->where('check_in', '<', $request->check_out)
             ->where('check_out', '>', $request->check_in);
       })->exists();
   // ↑ Proper date range overlap: existing_start < new_end AND existing_end > new_start
   
   Example Problem Scenario:
   - Existing booking: Feb 5-10 (property occupied Feb 5-9, available Feb 10)
   - New booking request: Feb 10-15
   
   Old Logic Result: CONFLICT (WRONG) - Feb 10 was incorrectly flagged
   New Logic Result: OK (CORRECT) - Feb 10-15 is allowed
   
   Impact: CRITICAL - Users were unable to book available dates, blocking
   revenue. Valid bookings were being rejected, degrading user experience.
