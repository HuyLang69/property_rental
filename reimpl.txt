1. OVERVIEW - NEW BOOKING FEATURE
================================================================================

The booking feature allows authenticated users to:
1. Browse available listings
2. Select check-in/check-out dates
3. Confirm booking (simplified: no payment processing)
4. View their confirmed bookings in a dashboard
5. Cancel their own bookings
6. See booking history

KEY CHANGES FROM OLD IMPLEMENTATION:
- Remove payment form complexity (keep it ultra-simple for MVP)
- Merge payment + confirmation into single booking confirmation
- Add proper dashboard routes for viewing bookings
- Implement proper authorization checks
- Clean separation: Booking creation, storage, retrieval, cancellation

  cleaning_fee    BIGINT UNSIGNED NOT NULL (in cents)
  service_fee     BIGINT UNSIGNED NOT NULL (in cents)
  total           BIGINT UNSIGNED NOT NULL (in cents)
  
  status          ENUM('pending', 'confirmed', 'cancelled') DEFAULT 'pending'

  updated_at      TIMESTAMP


INDEXES:
  - user_id (guests find their bookings quickly)
  - listing_id (hosts check conflicts quickly)
  - status (filter by confirmed/pending/cancelled)

3. ELOQUENT MODEL
================================================================================

File: app/Models/Booking.php

Key Elements:

A. Relationships:
   - belongsTo(User, 'user_id') as 'guest'
   - belongsTo(Listing, 'listing_id') as 'property'
   - hasOne(Review, 'booking_id') as 'review'

B. Fillable Fields:
   user_id, listing_id, check_in, check_out, guests,
   price_per_night, cleaning_fee, service_fee, total, status

C. Casts:
   check_in  → 'date'
   check_out → 'date'
   status    → 'string'

D. Accessors (NOT mutators, read-only):

   getNightsAttribute(): int
   - Returns: Carbon check_in.diffInDays(check_out)
   - Usage in views: $booking->nights

   getServiceFeeInDollarsAttribute(): string
   - Returns: number_format(service_fee / 100, 2)
   - Usage: $booking->service_fee_in_dollars

   getTotalInDollarsAttribute(): string
   - Returns: number_format(total / 100, 2)

E. Scopes (chainable query builders):

   scopeConfirmed($query): Builder
   - Returns: $query->where('status', 'confirmed')
   - Usage: Booking::confirmed()->get()

   scopePending($query): Builder
   - Returns: $query->where('status', 'pending')

   scopeByGuest(User $user): Builder
   - Returns: $query->where('user_id', $user->id)->latest()

   scopeByListing(Listing $listing): Builder
   - Returns: $query->where('listing_id', $listing->id)->latest()

   scopeConflictingDates(Listing $listing, $checkIn, $checkOut): Builder
   - Returns confirmed bookings that overlap requested dates
   - WHERE: check_in < $checkOut AND check_out > $checkIn
   - Usage: $conflicts = Booking::conflictingDates($listing, $in, $out)->exists();

F. Methods:

   canBeCancelled(): bool
   - Returns: $this->status !== 'cancelled' && $this->check_in > now()

   cancel(): bool
   - Sets status = 'cancelled'
   - Returns: $this->update(['status' => 'cancelled'])

   isOwnedBy(User $user): bool
   - Returns: $this->user_id === $user->id

4. CONTROLLER
================================================================================

File: app/Http/Controllers/BookingController.php

Middleware: All actions require 'auth'

A. create(Request $request): View
   ────────────────────────────
   - GET /bookings/create?listing_id=X&check_in=...&check_out=...&guests=N
   
   Validates input:
     listing_id  required|exists:listings,id
     check_in    required|date|after_or_equal:today
     check_out   required|date|after:check_in
     guests      required|integer|min:1|max:10
   
   Checks:
     - User can't book own listing
     - guests ≤ listing->max_guests
     - No date conflicts (scoped to confirmed bookings only)
   
   Returns view with calculated prices:
     - nights
     - subtotal (price_per_night × nights)
     - cleaning_fee
     - service_fee (12% of subtotal)
     - total (all three fees combined)


B. store(Request $request): RedirectResponse
   ────────────────────────
   - POST /bookings
   
   Validates same as create() + card fields (format validation only, not stored):
     card_number   required|regex:/^\d{4}\s\d{4}\s\d{4}\s\d{4}$/
     expiry        required|regex:/^\d{2}\s\/\s\d{2}$/
     cvv           required|digits_between:3,4
     cardholder    required|string
   
   And billing fields:
     billing_name, billing_address, billing_city, billing_zip, billing_country
   
   Server-side re-validation:
     - Repeat all date/guest checks (never trust client)
     - Reconfirm conflict check
     - Calculate fresh totals (never use client-submitted amounts)
   
   Creates booking:
     - status = 'confirmed' (simplified: no pending state)
     - All prices captured as snapshots
     - CSRF token validated automatically
   
   Returns: redirect()->route('bookings.show', $booking->id)
            ->with('success', 'Booking confirmed!')


C. show(Booking $booking): View
   ────────────────────────────
   - GET /bookings/{id}
   
   Authorization (abort if unauthorized):
     - User must be: booking owner OR listing host
   
   Loads relationships:
     - booking.listing.coverImage
     - booking.listing.host
   
   Returns view with all booking + pricing details


D. cancel(Booking $booking): RedirectResponse
   ────────────────────────────
   - PATCH /bookings/{id}/cancel
   
   Authorization: User must own the booking
   
   Checks:
     - Booking not already cancelled
     - Check-in date still in future (no past cancellations)
   
   Updates: status = 'cancelled'
   
   Returns: redirect()->route('dashboard.trips')
            ->with('success', 'Booking cancelled')


E. index() [OPTIONAL - list user's bookings]
   ────────────────
   - GET /dashboard/trips (or here if preferred)
   
   Returns all bookings for logged-in user sorted by check_in DESC

5. ROUTES
================================================================================

File: routes/web.php

Replace booking section with:

```php
// ── Bookings (auth only) ───────────────────────────────────
Route::middleware('auth')->group(function () {
    // Browse + confirm
    Route::get('/bookings/create', [BookingController::class, 'create'])
        ->name('bookings.create');
    Route::post('/bookings', [BookingController::class, 'store'])
        ->name('bookings.store');
    
    // View single booking
    Route::get('/bookings/{booking}', [BookingController::class, 'show'])
        ->name('bookings.show');
    
    // Cancel booking
    Route::patch('/bookings/{booking}/cancel', [BookingController::class, 'cancel'])
        ->name('bookings.cancel');
    
    // Dashboard: list user's bookings
    Route::get('/dashboard/trips', [BookingController::class, 'index'])
        ->name('dashboard.trips');
    
    // Alternative name for same route (for flexibility)
    Route::get('/dashboard/bookings', [BookingController::class, 'index'])
        ->name('dashboard.bookings');
});
```

Key points:
- All routes require 'auth' middleware
- Consistent naming: bookings.* for main actions
- dashboard.* for dashboard views
- POST always uses route name 'bookings.store', not separate endpoints

7. VALIDATION & BUSINESS LOGIC
================================================================================

Date Conflict Detection:
──────────────────────

Use simple but correct logic:

```php
$conflict = Booking::where('listing_id', $listing->id)
    ->confirmed()
    ->where(function ($q) use ($checkIn, $checkOut) {
        $q->where('check_in', '<', $checkOut)
          ->where('check_out', '>', $checkIn);
    })
    ->exists();

if ($conflict) {
    return back()->withErrors([
        'check_in' => 'These dates are not available.'
    ]);
}
```

Explanation:
- Only confirmed bookings block dates (pending are ignored)
- Two intervals don't overlap IFF one ends before the other starts
- Negative: NOT (A.end ≤ B.start OR A.start ≥ B.end)
- Positive: A.start < B.end AND A.end > B.start


Max Guests Check:
───────────────

```php
if ($guests > $listing->max_guests) {
    return back()->withErrors([
        'guests' => "Maximum {$listing->max_guests} guests allowed."
    ]);
}
```


Price Calculation (server-side only):
────────────────────────────────────

```php
$checkIn = Carbon::parse($request->check_in);
$checkOut = Carbon::parse($request->check_out);

$nights = $checkIn->diffInDays($checkOut); // int

$subtotal = $listing->price_per_night * $nights; // cents
$cleaning = $listing->cleaning_fee;              // cents
$serviceFee = (int) ($subtotal * 0.12);          // 12% of subtotal
$total = $subtotal + $cleaning + $serviceFee;    // all in cents

// Store all four values in booking for historical accuracy
```

Authorization:
───────────────

Booking owner: $booking->user_id === Auth::id()
Listing owner: $booking->listing->user_id === Auth::id()

User can view booking if they're either owner or host.
User can only cancel bookings they own.

10. COMMON PITFALLS TO AVOID
================================================================================

❌ WRONG: Using includeBetween for date overlap
   Problem: Allows adjacent dates to conflict
   Solution: Use < and > as shown above

❌ WRONG: Trusting client-submitted totals
   Problem: Users can manipulate prices
   Solution: Recalculate on server in store() method

❌ WRONG: Forgetting @csrf on forms
   Problem: 419 Token Mismatch error
   Solution: ALWAYS include @csrf in POST forms

❌ WRONG: Missing route names for navigation
   Problem: "Route not defined" errors
   Solution: Define both dashboard.bookings and bookings.show

❌ WRONG: Storing actual credit card details
   Problem: PCI compliance violation
   Solution: Only validate format, never store card number

Questions to clarify before starting:
1. Should pending bookings exist, or go straight to confirmed?
2. Should there be actual payment processing (Stripe) or just form validation?
3. Should cancellations be instant or require approval?
4. Should reviews require a completed booking?

Current design assumes:
- Bookings go straight to "confirmed" (no pending state used)
- Payment is form validation only (no charge processing)
- Cancellations are instant (if check_in is future)
- Reviews can be left only on confirmed bookings
